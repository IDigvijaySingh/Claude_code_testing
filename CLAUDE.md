# CLAUDE.md

This file defines how Claude Code should operate in this repository. The `@claude` GitHub Action (see `.github/workflows/claude.yml`) and any local Claude Code sessions should follow these conventions.

## Repository purpose

This is a sandbox repository for testing and demonstrating Claude Code behavior on GitHub issues and pull requests. There is no application stack yet — when one is added, update the relevant sections below.

## How to invoke Claude

- **On an issue:** include `@claude` in the issue body or in a comment to trigger the workflow.
- **On a pull request:** include `@claude` in a PR review comment to trigger the workflow.
- The workflow runs `anthropics/claude-code-action@v1` and requires the `ANTHROPIC_API_KEY` repository secret.

## Branch and PR conventions

- **Default branch:** `main`.
- **Claude-authored branches:** named `claude/issue-<number>-<YYYYMMDD>-<HHMM>` (auto-generated by the action when triggered on an issue).
- **Direct pushes to `main`:** not allowed. All changes land via pull request.
- **PR titles:** short, imperative, lowercase except for proper nouns. Example: `add gitignore and readme`.
- **PR descriptions:** include a short summary, a reference to the originating issue (`Closes #N` when appropriate), and a test plan if any runnable code changed.
- **Commits:** small, focused, with imperative subject lines. Co-author trailers are encouraged when work was paired with `@claude`.

## What Claude should do

- Read this `CLAUDE.md` at the start of every session.
- Read the issue or PR thread carefully before acting. The instructions live in the comment that mentions `@claude`; other comments are context only.
- When a request is ambiguous, ask for clarification in the GitHub comment rather than guessing.
- Prefer the smallest change that satisfies the request. Do not refactor, restructure, or add scaffolding that wasn't asked for.
- Keep responses concise. Update the existing tracking comment rather than posting new ones.

## What Claude should NOT do

- Do not modify files under `.github/workflows/` — the GitHub App lacks the `workflows` permission and the push will be rejected.
- Do not commit secrets, API keys, `.env` files, or anything matching the patterns in `.gitignore`.
- Do not force-push, rewrite history, or delete branches.
- Do not open a PR against `main` without first pushing the branch and providing a manual PR-creation link.
- Do not add a language stack, framework, or build tooling without being explicitly asked.

## Review checklist

When asked to review a PR, Claude should check at minimum:

- Does the change match the stated intent of the issue/PR description?
- Are there obvious bugs, race conditions, or unhandled error paths?
- Are there security concerns (injection, secret leakage, unsafe deserialization, overly broad permissions)?
- Is the diff minimal — no unrelated reformatting or drive-by changes?
- Are tests present and meaningful when code behavior changes?
- Is documentation (including this file) updated when conventions change?
